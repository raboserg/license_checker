Правильное использование CMAKE _ * _ OUTPUT_DIRECTORY

ПРЕДИСЛОВИЕ: я только говорить о компиляции локально, не об установке проектов. Это потому, что я недостаточно исследовал install с CMake, пожалуйста, не стесняйтесь, если мой вопрос имеет непосредственное отношение к install практики (кажется вероятным).

TL; DR

  1. В каких сценариях ты не Хотите собрать все библиотеки проектов, которые собираются в одном каталоге? Почему никто никогда CACHE CMAKE_*_OUTPUT_DIRECTORY пути?

  2. Есть ли необходимость выполнять $<CONFIG> спецификации уровня напрямую?

  3. Должно ли общее значение по умолчанию CMAKE_BINARY_DIR, CMAKE_CURRENT_BINARY_DIR, или же PROJECT_BINARY_DIR?

1. Кешировать или не кешировать?

От этот отличный ответ

set(CMAKE_ARCHIVE_OUTPUT_DIRECTORY ${CMAKE_BINARY_DIR}/lib)
set(CMAKE_LIBRARY_OUTPUT_DIRECTORY ${CMAKE_BINARY_DIR}/lib)
set(CMAKE_RUNTIME_OUTPUT_DIRECTORY ${CMAKE_BINARY_DIR}/bin)

который вы видите во многих проектах в Интернете.

  • Родитель CMakeLists.txt не могу переопределить это.
  • Если родительский проект хочет / должен изменить их, например, положить все в одну папку, невозможно.

Таким образом, предлагаемый пересмотр будет всегда CACHE PATH "description":

set(CMAKE_ARCHIVE_OUTPUT_DIRECTORY ${CMAKE_BINARY_DIR}/lib CACHE PATH "Where to place compiled static libraries.")
set(CMAKE_LIBRARY_OUTPUT_DIRECTORY ${CMAKE_BINARY_DIR}/lib CACHE PATH "Where to place compiled shared libraries.")
set(CMAKE_RUNTIME_OUTPUT_DIRECTORY ${CMAKE_BINARY_DIR}/bin CACHE PATH "Where to place compiled executables.")
  • есть ли смысл не кешировать?

2. Лечение $<CONFIG> строит

Это должно в значительной степени всегда быть оставленным генератору, да? Мое понимание (и опыт) заключается в настройке неконфигурирующего генератора (Visual Studio, XCode, …), использующего папки для каждой конфигурации, все еще с содержимым для размещения вещей там, где они обычно находятся. AKA, не устанавливая уровень конфигурации CMAKE_*_OUTPUT_DIRECTORY_* Я передаю управление генератору, чтобы он решил.

3. BINARY_DIR, CURRENT_BINARY_DIR, или же PROJECT_BINARY_DIR?

С другой стороны, скажем, разработчик делает не хотите, чтобы другие проекты, которые они создают, попали в тот же каталог. Разве это не должно быть в дальнейшем, что я использую CMAKE_CURRENT_BINARY_DIR ИЛИ ЖЕ PROJECT_BINARY_DIR? В том случае, если просто сырье CMAKE_BINARY_DIR используется, без CACHEтогда у меня нет абсолютно никакой возможности предотвращать подпроект, который я строю, заканчивая рядом с моим.

Используя либо CMAKE_CURRENT_BINARY_DIR или же PROJECT_BINARY_DIRесли пользователь делает не хотите, чтобы эта библиотека оказалась рядом с родительским проектом, они могут просто установить CMAKE_*_OUTPUT_DIRECTORY переменные после настраивая этот.

Резюме

По сути, нет никаких стандартов для того, как эти переменные должны использоваться. Я в восторге от того, насколько универсален CMake, я не говорю, что они вообще должны делать какие-либо значения по умолчанию — это должно быть определено проектом. Я пытаюсь понять, что является наиболее подходящим выбором по умолчанию, что также позволяет разработчикам обойти мои значения по умолчанию, если они того пожелают.

Решение

Я с @ Цываревым. Я не вижу пользователя проекта с поддержкой CMake, желающего изменить путь вывода артефактов сборки, только путь установки.

1. Кешировать или не кешировать?

Просто не устанавливайте CMAKE_*_OUTPUT_DIRECTORY переменные как кэшированные, потому что, если другой проект CMake использует ваш как подпроект, вы при определенных обстоятельствах перезапишете настройки вашего родительского проекта (кэшированные переменные являются глобальными).

2. Лечение $<CONFIG> строит

Да. Он дается только тогда, когда значения по умолчанию генератора не соответствуют («Имя конфигурации» = «Имя подпапки»).

3. CMAKE_BINARY_DIR, CMAKE_CURRENT_BINARY_DIR, или же PROJECT_BINARY_DIR?

Так что я бы сказал, что вариант сохранения будет сначала проверить с if () заявление, если кто-то уже установил переменную, а также использовать PROJECT_BINARY_DIR:

if (NOT CMAKE_ARCHIVE_OUTPUT_DIRECTORY)
set(CMAKE_ARCHIVE_OUTPUT_DIRECTORY "${PROJECT_BINARY_DIR}/lib")
endif()

Тогда пользователь вашего проекта может по-прежнему устанавливать переменные извне или родительский проект может установить его, и у вас есть значение по умолчанию, если оно не было установлено.
